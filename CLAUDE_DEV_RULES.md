## 1. 数据展示规则

### 1.1 数字精度
- 所有数字字段（数量、吨数、单价等）显示时小数保留 **2 位**
- `fmtQty` 使用 `toFixed(2)`，输入 `step="0.01"`
- 金额字段使用 `fmtMoney`，同样保留 2 位

### 1.2 必填标识
- 必填字段的红色 `*` 号：**字段有值后隐藏**，无值时才显示
- 模式：`{!value && <span className="text-rose-400">*</span>}`
- 表格列头（多行共用）的 `*` 可保留始终显示
- **Why:** 用户填写后 `*` 不再有意义，减少视觉噪音
- **How to apply:** 每个 `*` 找到对应 state 变量，用 `{!stateVar && <span>...</span>}` 包裹

## 2. 交互规则

### 2.1 单选/下拉默认值
- 所有单选下拉（SearchableSelect、select、radio）**不设默认选择**
- `useState` 初始值统一为 `""`（空字符串）
- 日期类字段可保留今天的默认值
- **Why:** 避免用户忘记选择而提交错误数据

### 2.2 选择项取消选择
- 所有选择组件必须支持**取消选择**（清空已选项）
- SearchableSelect：点击已选中项可取消选择，或提供清除按钮
- **Why:** 用户可能选错需要回退

### 2.3 拖拽排序
- 支持排序的列表（品牌、仓库、型号等）统一使用原生 HTML5 Drag & Drop
- 排序间距策略：`(index + 1) * 10`，新项自动排最后 `max_order + 1`
- 后端需配套 `sort_order` 字段 + `PATCH /reorder` 接口
- **Why:** 用户需要自定义数据展示顺序

## 3. 数据安全规则

### 3.1 引用保护
- 被其他业务单据引用的基础数据（品牌、仓库、型号等）**禁止删除**，只能修改
- 三层防线：数据库 `UniqueConstraint` → 服务层检查 → 前端隐藏删除按钮
- 检查范围覆盖：采购合同、销售合同、发货计划
- **Why:** 防止数据不一致，保留业务追溯链

### 3.2 唯一性校验
- 名称类字段（品牌名、撮合平台名、仓库名、型号名等）**必须做重名校验**
- 三层：数据库唯一约束 → 服务层查重 → 前端错误提示
- **Why:** 避免重复录入造成数据混乱

## 4. 开发规范

### 4.1 技术规范
- 所有回复和交互使用**中文**
- 所有代码注释使用**中文**
- 严格遵循 **SOLID** 原则
- 优先使用异步 `async/await`
- 所有结构变更必须通过 **Alembic** 迁移
- 每个文件写完立即 **mypy** 类型检查
- 禁止大范围重构、一次开发多个模块、为未来需求提前实现功能（**YAGNI**）

### 4.2 模块结构（Vertical Slice）
```
modules/{module}/
├── models.py      # ORM 模型
├── schemas.py     # Pydantic 校验
├── repository.py  # 数据访问层
├── service.py     # 业务逻辑层
├── router.py      # API 路由
└── events.py      # 事件（可选）
```

### 4.3 模块间通信
- `Module A → EventBus → Module B` ✅
- `Module A → Module B.Service` ❌

### 4.4 错误处理
- 业务异常继承 `DomainError`
- `ConflictError` → HTTP 409（重复/冲突）
- `NotFoundError` → HTTP 404
- 前端统一通过 `handleResponse` 提取 `body.message` 显示错误

---

## 5. 架构规则

- **DDD Lite + Vertical Slice**: 每个业务模块独立目录
- **Event Bus**: 跨模块通信必须通过事件总线
- **Shared Kernel**: 共享基础类放在 `app/shared/`
- **Business Partner**: 联系人、地址、银行等拆分独立表
